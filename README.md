### 프로젝트 개요

<img width="607" height="312" alt="image" src="https://github.com/user-attachments/assets/f0cad142-381a-4aad-855d-c9404d3a0fc5" />

- 전자부품 유통사의 업무 담당자가 **자연어 질문만으로** 재고·매출·매입 데이터를 즉시 조회하고 인사이트를 얻을 수 있는 기업 AI 비서 서비스입니다.
- 기존 챗봇은 내부 DB와 직접 연결되지 않아 실무 데이터 조회가 불가능했고, 대화 맥락 유지와 완전 자동화에 한계가 있었습니다.
- **LLM + RAG + Orchestration** 세 가지를 결합하여 이러한 한계를 극복했습니다.

![image](여기에_이미지_링크_붙여넣기)
> PPT 6페이지 — LLM · RAG · Orchestration 관계도

- LLM의 부정확성은 RAG가 보완하고, RAG의 검색 의존성은 Orchestration이 통제하며, LLM의 비용·성능 문제는 Orchestration이 모델 전략으로 조정합니다.
<br><br>

### LLM 서버 상세

> FastAPI 기반의 에이전트 파이프라인으로, 자연어 질문을 SQL로 변환하여 DB를 조회하고 자연어 답변을 생성합니다.

#### 전체 플로우

![image](여기에_이미지_링크_붙여넣기)
> PPT 13페이지 — Agent Graph 노드 흐름

사용자의 질문은 아래 순서로 처리됩니다.

- **Memory** — 이전 대화 기록을 불러와 맥락을 주입합니다. 후속 질문을 완전한 질문으로 재조립합니다.
- **Refine** — 제조사·고객사 이름 오타를 fuzzy 매칭으로 교정하고, 파트넘버 오타를 벡터 검색으로 보정합니다.
- **Router** — 키워드 매칭과 LLM fallback으로 질문 유형을 분류합니다. `INVENTORY` 질의는 SQL 생성으로, `CHIT_CHAT` / `TECH_SALES`는 바로 답변 생성으로 이동합니다.
- **SQL Gen** — fewshot 예시, 비즈니스 용어, 스키마 정보를 RAG로 검색하여 LLM이 SQL을 생성합니다.
- **Execute DB** — 보안·문법·논리·스키마·JOIN 규칙 5단계 검증 후 PostgreSQL에 실행합니다.
- **Validate** — NULL 비율, 음수 매출, 극단값을 검사합니다. 이상 감지 시 SQL Gen으로 재시도합니다.
- **Answer** — 실제 DB 조회 결과를 FACT로 LLM에 주입하여 자연어 답변을 생성합니다.
- **Save** — 질문·답변·SQL·처리시간을 DB에 저장하여 다음 대화의 기억으로 활용합니다.

> 재시도는 최대 3회까지 수행되며, 초과 시 에러 메시지를 안내합니다.
<br><br>

#### 핵심 설계 포인트

- **하이브리드 RAG 검색** — 벡터 유사도(60%) + BM25 키워드(40%) 검색을 병합한 후 Cohere로 재정렬합니다. 개별 전략 실패 시 자동 fallback하여 서비스가 중단되지 않습니다.
- **SQL 안전 검증** — `SELECT *` 금지, 존재하지 않는 테이블·컬럼 차단, Cartesian Product 감지, 허용된 JOIN 조합 강제 등 5단계 정적 검증을 수행합니다.
- **에러 기반 재시도** — DB 실행 실패 시 에러 메시지를 분석하여 수정 힌트를 생성하고 프롬프트에 삽입합니다. LLM이 같은 실수를 반복하지 않도록 설계했습니다.
- **Hallucination 방지** — 답변 생성 시 실제 DB 조회 결과만을 FACT로 명시하여 없는 데이터를 지어내는 것을 방지합니다.
- **유연한 모델 교체** — LLM, 임베딩, 리랭커 각각 추상 인터페이스로 분리되어 있어 `dependency.py` 한 곳만 수정하면 다른 모델로 교체 가능합니다.

### 모델 선정 과정

#### LLM 비교

![image](여기에_이미지_링크_붙여넣기)
> PPT 15페이지 — LLM 응답시간 및 속도 비교

- 무료 모델 3종을 응답 시간과 속도 기준으로 비교하였습니다.
- Groq-Llama-3.1-8B가 속도(583.4 tokens/s)는 가장 빠르나 SQL 생성 정확도에 한계가 있었고, Gemini-2.5-Flash는 응답 시간이 길지만 정확도와 한국어 이해도가 우수하여 최종 선정하였습니다.
<br><br>

#### 임베딩 모델 비교

![image](여기에_이미지_링크_붙여넣기)
> PPT 17페이지 — 임베딩 모델 사양 비교

![image](여기에_이미지_링크_붙여넣기)
> PPT 18페이지 — 문장 유사도 테스트 결과

- 4종의 임베딩 모델을 논리 변별력과 맥락 파악력 기준으로 비교하였습니다.
- 문장 유사도 테스트에서 요약 정확도(0.67)와 논리 정확도(0.56) 모두 준수한 Gemini embedding-v1을 선정하였습니다.
<br><br>

### 향후 개선 방안

- 성공한 SQL을 자동으로 fewshot 데이터로 축적하여 RAG 검색 품질을 지속적으로 개선합니다.
- 맥락 기억 기능을 계층화하여 장기 기억과 단기 기억을 분리하고 성능을 고도화합니다.
- 다양한 임베딩 모델 추가 테스트를 통해 더 나은 검색 품질을 확보합니다.
- 제품 재고 부족 시 자동 알림 메시지 기능을 개발합니다.
- 챗봇 답변 시각화 기능(차트, 그래프)을 추가합니다.
- 데이터 축적 이후 LOCAL LLM 도입을 재시도합니다.
